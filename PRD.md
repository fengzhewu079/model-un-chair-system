# Product Requirements Document (PRD)
# Model UN Chair System

---

## 1. 产品概述

### 1.1 产品定位

Model UN Chair System 是一个专为模拟联合国会议主席设计的网页端会议指挥工具.

**核心定位**: 将高度依赖经验和记忆的主席工作, 转化为结构化, 可控, 可留痕的会议操作流程.

**明确边界**:
- 这是 Chair 端工具, 不是代表端系统
- 不替代会议规则或主席判断, 而是辅助执行
- 不做会议规则培训, 假设 Chair 已理解基本议事规则

### 1.2 目标用户

**主要用户**: 模拟联合国会议主席(Chair)

**用户画像**:
- 年龄: 高中生和大学生
- 经验: 新手和经验丰富的 Chair 都有
- 技术能力: 有使用电脑的习惯, 但不应有高技术门槛
- 工作状态: 会议中需要处理多线程任务, 但信息输入压力不大
- 痛点: 文档记录速度慢, 容易出错, 不够专业

**使用环境**:
- 设备: 笔记本电脑
- 网络: 需要稳定网络连接
- 会场: 会投屏显示(投屏为操作界面镜像)
- 时长: 单次会议 3-4 小时
- 规模: 10-100+ 人不等

**MVP 关键设计决策**:
- ✓ 界面语言: 全英文
- ✓ 视觉主题: 浅色主题(适合投影)
- ✓ 投屏方式: 操作界面镜像(只需一套界面)
- ✓ 交互方式: 纯鼠标点击(无快捷键)
- ✓ 代表管理: 手动输入(无预设名单)
- ✓ 时间提示: 10 秒警报(视觉 + 音频)

## 2. 问题定义与核心价值

### 2.1 核心痛点

#### 痛点1: Chair 的工作高度依赖个人经验与记忆
- 现状: 使用文档记录, 信息高度分散
- 后果: 容易遗漏, 出错, 速度跟不上
- 严重性: 生死级 - 直接影响会议流程和公平性

#### 痛点2: 会议节奏一快, 错误成本极高
- 现状: 发言顺序, 动议处理, 表决步骤一旦出错引发争议
- 后果: 影响会议公平性和 Chair 的专业性感知
- 严重性: 生死级 - 损害会议信任度

#### 痛点3: 会议过程缺乏结构化留痕
- 现状: 无法完整回溯关键流程和决策节点
- 后果: 赛后复盘, 培训, 申诉缺乏可靠依据
- 严重性: 效率级 - 影响会议质量提升

#### 痛点4: 新手 Chair 学习成本高且不稳定
- 现状: 不同 Chair 水平差异大, 会议质量不可控
- 后果: 组织方对会议结果不可控, 参会体验波动大
- 严重性: 效率级 - 影响会议标准化

### 2.2 现有解决方案分析

| 解决方案 | 方式 | 痛点 | 为什么不够 |
|---------|------|------|-----------|
| 纸笔记录 | 手写记录发言者, 动议, 时间 | 速度慢, 难查找, 无法回溯 | 高强度会议下完全跟不上 |
| 文档(Word/Excel) | 电子表格记录流程 | 需要手动维护状态, 无结构化流程 | 依然依赖记忆, 无法自动提示 |
| 纯记忆 | 凭经验记住所有信息 | 完全依赖个人能力, 极易出错 | 新手无法使用, 老手也不稳定 |

**核心问题**: 现有方案都是"记录工具", 而不是"流程执行工具". Chair 需要自己维护会议状态, 自己判断下一步, 自己记住所有信息.

### 2.3 核心价值主张

> **将 Chair 的会议管理工作从"靠记忆和经验"转化为"基于规则的流程执行"**

**具体价值**:
1. 降低 Chair 的认知负担, 让其专注于判断而非记录
2. 提升会议流程的清晰度, 稳定性和可验证性
3. 通过工具而非培训, 缩小不同 Chair 之间的能力差距
4. 让会议过程可回溯, 可复盘, 可申诉

### 2.4 差异化定位

| 维度 | 现有方案 | 本产品 |
|------|---------|--------|
| 状态维护 | 人脑记忆 | 系统自动维护 |
| 流程引导 | 依赖经验 | 系统结构化引导 |
| 错误预防 | 事后发现 | 事前阻止或提示 |
| 可追溯性 | 无或手动 | 自动记录全流程 |
| 学习成本 | 需大量培训和实战 | 降低门槛, 工具化引导 |

**差异化本质**: 从"人脑维护会议状态"到"系统维护会议状态, 人只做判断和确认".

**可感知性**: Chair 能明显感受到从"我得记住所有信息"到"我只需要看屏幕操作"的转变.

**影响决策**: 如果系统可靠, Chair 会从依赖记忆转为依赖系统.

**护城河**: 需要深度理解 MUN 议事规则, 以及 Chair 的真实工作流程.

## 3. 核心假设(待验证)

### 假设1: 统一指挥界面能减少流程性错误
- **假设内容**: 如果 Chair 拥有一个统一的会议指挥界面, 会议中流程性错误会显著减少
- **验证方式**: 对比使用前后的错误率(漏掉发言者, 动议处理错误, 表决计算错误等)
- **成功标准**: 错误率下降 50% 以上
- **失败标准**: 错误率没有明显变化, 或反而增加

### 假设2: 结构化呈现能提高决策稳定性
- **假设内容**: 如果发言, 动议和表决被结构化呈现, Chair 在高压环境下的决策稳定性会提高
- **验证方式**: Chair 主观反馈 + 观察者评分(会议节奏是否平稳, 处理是否果断)
- **成功标准**: Chair 感到更自信, 处理更流畅
- **失败标准**: Chair 感到更混乱, 或需要频繁脱离系统手动处理

### 假设3: 自动留痕能提升专业性感知
- **假设内容**: 如果会议过程可以被自动留痕和导出, 会议的专业性和公平性感知会提升
- **验证方式**: 代表和组织方的主观评价
- **成功标准**: 明显提升专业性评价
- **失败标准**: 无明显变化, 或代表质疑系统的公正性

### 假设4: 工具引导能被 Chair 接受
- **假设内容**: 如果工具能够引导而非替代规则执行, Chair 会愿意在真实会议中持续使用该系统
- **验证方式**: 实际使用率 + 重复使用意愿
- **成功标准**: Chair 在真实会议中持续打开并使用系统, 且愿意推荐给其他 Chair
- **失败标准**: Chair 试用后放弃, 或只在练习时用, 真实会议时不敢用

### 假设5: Roll Call 结构化能提升会议规范性
- **假设内容**: 如果 Roll Call 过程被结构化并可视化, Chair 会更愿意严格执行会前流程, 且会议中关于表决合法性的争议会减少
- **验证方式**: Chair 执行 Roll Call 的比例 + 表决争议发生率
- **成功标准**: Chair 愿意在每次会议前执行 Roll Call, 且表决争议明显减少
- **失败标准**: Chair 跳过 Roll Call 直接开会, 或 Roll Call 数据在表决时未被使用

## 4. MVP 定义与功能范围

### 4.1 MVP 核心目标

**唯一核心假设**: 如果 Chair 拥有一个统一的会议状态管理界面, 能够无需记忆地处理发言, 动议和表决, 那么 Chair 在真实会议中会愿意持续使用它, 且流程性错误会显著减少.

**验证方式**: 至少 5 位不同经验水平的 Chair 在真实会议中使用, 收集错误率数据和主观反馈.

### 4.2 功能分层

#### Core (MVP 必须包含)

**0. Roll Call (会前点名)**
- 作为 Chair, 我需要在会议开始前确认代表出席状态, 以便建立表决基数和会议合法性基础
- 功能:
  - 代表列表输入(批量粘贴或单个添加)
  - 逐个点名并标记出席状态:
    - Present (出席, 可在程序性表决中弃权)
    - Present and Voting (出席且投票, 不可弃权)
    - Absent (缺席)
  - 实时统计:
    - Total Delegates
    - Present
    - Present and Voting
    - Absent
  - 完成并锁定 Roll Call, 作为后续表决的基数
  - Roll Call 结果自动记录, 可导出
- 价值:
  - 避免人工点名遗漏或记录错误
  - 为后续表决提供准确的票数基数
  - 防止表决合法性争议
  - 让会议在规范状态下开始

**1. 会议状态管理**
- 作为 Chair, 我需要系统明确显示当前会议处于什么阶段, 以便我知道现在该做什么
- 支持的状态:
  - General Speakers List (GSL)
  - Moderated Caucus
  - Unmoderated Caucus
  - Voting Procedure
  - Suspension of Meeting
- 状态切换必须符合议事规则(例如: 不能从 Voting 直接跳到 Unmoderated)
- 每次状态切换自动记录时间戳

**2. 发言队列管理**
- 作为 Chair, 我需要快速记录谁要发言, 以便不漏掉任何代表
- 功能:
  - 添加发言者到队列:
    - 搜索栏输入(输入 2 个字母后自动弹出建议)
    - 智能建议列表(仅显示 Roll Call 中出席的代表)
    - 可直接输入完整名称或点击建议选择
    - 允许添加不在 Roll Call 列表中的代表(处理特殊情况)
  - 显示当前发言者(高亮显示)
  - 显示等待队列(按顺序排列)
  - 移除队列中的发言者(代表放弃发言)
  - 调整发言顺序(拖拽或上下移动)
- 发言计时:
  - 为当前发言者启动倒计时
  - 时间到达时明显提示(视觉+音频)
  - 可暂停/继续/重置计时
  - 可手动结束发言并自动切换到下一位
- 不同发言模式:
  - GSL 模式: 固定时长(如 1 分钟), 队列可持续添加
  - Moderated Caucus 模式: 固定单次发言时长 + 总时长限制

**3. 动议处理**
- 作为 Chair, 我需要系统帮我记录和处理动议, 以便不遗漏任何动议且处理顺序正确
- 功能:
  - 记录动议信息:
    - 动议类型(Motion for Moderated Caucus, Motion to Close Debate, Motion to Adjourn 等)
    - 提出者
    - 具体参数(如 Moderated Caucus 的总时长和单次发言时长)
  - 动议状态管理:
    - Pending (等待处理)
    - In Voting (正在表决)
    - Passed (通过)
    - Failed (未通过)
  - 流程引导:
    - 提示 Chair 询问附议(如需要)
    - 引导进入表决流程
    - 表决通过后自动切换会议状态(如进入 Moderated Caucus)
  - 动议优先级:
    - 如果有多个动议, 按优先级排序(特权动议 > 程序动议)
    - 提示 Chair 应该先处理哪个

**4. 表决管理**
- 作为 Chair, 我需要快速准确地统计表决结果, 以便不出错且提高效率
- 功能:
  - 记录票数:
    - 赞成票数 (For)
    - 反对票数 (Against)
    - 弃权票数 (Abstain)
  - 输入方式:
    - 手动输入数字
    - 或点击按钮递增/递减
  - 自动计算(基于 Roll Call 结果):
    - 有效票数 = 赞成 + 反对 (弃权不计入)
    - 简单多数阈值 = 有效票数 / 2 (向上取整)
    - 绝对多数阈值 = Present + Present and Voting 总数 * 2/3
    - 判断表决是否通过(根据动议类型自动应用规则)
    - 显示结果(通过 Pass / 未通过 Fail)
  - 表决历史:
    - 显示已完成的表决记录
    - 包括动议内容, 票数, 结果, 时间

**5. 计时功能**
- 作为 Chair, 我需要系统帮我管理时间, 以便不超时或混乱
- 功能:
  - 发言倒计时:
    - 显示剩余时间(分:秒格式)
    - 时间接近结束时变色提示(如最后 30 秒变红)
    - 时间到达时音频+视觉提示
  - 会议阶段倒计时:
    - Moderated Caucus 总时长倒计时
    - Unmoderated Caucus 时长倒计时
  - 计时控制:
    - 暂停/继续
    - 重置
    - 延长时间(如有需要)

**6. 基础会前设置**
- 作为 Chair, 我需要在会议开始前进行基础设置, 以便系统能正常运行
- 功能:
  - 输入会议名称
  - 输入代表列表(用于 Roll Call):
    - 批量粘贴(每行一个国家/代表团名称)
    - 或单个添加
  - 设置默认发言时长(如 GSL 每人 1 分钟)
  - 完成 Roll Call(点名确认出席)
- 暂不支持:
  - 标准国家名单模板(MVP 需手动输入)
  - 议题管理(MVP 单议题)
  - 自定义规则配置(MVP 按标准议事规则)

#### Later (验证通过后再做)

1. **会后数据导出和复盘**
   - 导出会议记录(PDF/Word)
   - 数据分析和可视化(发言分布, 动议统计等)
   - 会议录像/音频关联

2. **历史会议管理**
   - 保存多个会议记录
   - 会议模板复用
   - 会议对比分析

3. **高级会前设置**
   - 代表名单批量导入
   - 多议题管理
   - 自定义规则配置
   - 会议日程规划

4. **协作与分享**
   - 多 Chair 协作(主席+副主席)
   - 会议实时分享给观察员
   - 代表端显示(公开信息展示)

5. **智能辅助**
   - 动议类型智能推荐
   - 异常情况提示(如长时间无人发言)
   - 会议节奏分析和建议

6. **账号与权限**
   - 用户账号系统
   - 会议权限管理
   - 数据云端同步

#### Kill (明确不做)

1. **代表端功能**
   - 代表举手/申请发言
   - 代表端投票
   - 代表间通讯
   - 理由: 这是 Chair 工具, 不是代表端系统, 做代表端会极大增加复杂度且偏离核心价值

2. **会议规则培训**
   - 议事规则教学
   - Chair 培训课程
   - 理由: 假设 Chair 已理解基本规则, 这是培训系统而非操作系统

3. **会议内容管理**
   - 决议草案编辑
   - 文件共享
   - 理由: 已有专门工具(Google Docs 等), 不需要重复建设

4. **社交功能**
   - Chair 社区
   - 经验分享
   - 理由: 与核心价值无关, 属于社区产品

## 5. 用户故事与验收标准

### 5.1 核心用户故事

#### 故事0: 会前 Roll Call

**作为** Chair
**我想要** 在会议开始前快速完成 Roll Call 并确认出席状态
**以便** 建立准确的表决基数, 避免后续表决争议

**场景描述**:
1. Chair 打开系统, 批量粘贴代表名单(15 个国家)
2. 系统自动解析为代表列表, 按字母顺序显示
3. Chair 点击 "Start Roll Call"
4. Chair 逐个点名, 点击按钮标记状态: Present, Present and Voting, 或 Absent
5. 系统实时显示统计: Total: 15, Present: 12, Present and Voting: 10, Absent: 3
6. Chair 确认完成, 点击 "Complete Roll Call"
7. 系统锁定出席列表, 提示 "Roll Call Completed. Total Present: 12"
8. Chair 进入会议正式流程

**场景变化: 代表迟到**:
1. 会议进行中, 某代表迟到
2. Chair 在设置中点击 "Update Roll Call"
3. 将该代表状态从 Absent 改为 Present
4. 系统更新表决基数, 提示 "Voting base updated: 13 present"

**验收标准**:
- [ ] 可以批量粘贴代表列表, 系统正确解析(每行一个名称)
- [ ] 可以单击标记每个代表的出席状态
- [ ] 实时统计数据准确且显眼
- [ ] Complete Roll Call 后状态被锁定
- [ ] 可以在会中更新 Roll Call(需明确操作)
- [ ] Roll Call 结果作为表决基数, 自动用于后续表决计算

---

#### 故事1: 管理发言队列(使用智能搜索)

**作为** Chair
**我想要** 快速记录举手的代表并按顺序让他们发言
**以便** 不漏掉任何人且保持公平的发言顺序

**场景描述**:
1. 会议正在进行 GSL, 有 5 个代表陆续举手
2. Chair 点击 "Add Speaker" 输入框
3. Chair 输入 "US", 系统自动弹出建议:
   - United States of America
   - Russian Federation
4. Chair 点击 "United States of America", 代表被添加到队列
5. Chair 继续输入 "CH", 系统建议:
   - China
   - Chile
6. Chair 点击 "China", 添加到队列
7. Chair 用同样方式快速添加其余 3 个代表
8. Chair 点击 "Start Speaking" 让第一位代表发言, 系统自动开始计时
9. 第一位代表发言结束, Chair 点击 "Next", 系统自动切换到第二位
10. 如果某位代表临时放弃发言, Chair 点击删除按钮将其移除

**验收标准**:
- [ ] 输入 2 个字母后, 自动弹出智能建议列表
- [ ] 建议列表仅显示 Roll Call 中出席的代表
- [ ] 可以在 3 秒内添加一个发言者到队列(输入 + 选择)
- [ ] 允许输入不在 Roll Call 列表中的名称
- [ ] 当前发言者有明显视觉标识(如高亮, 放大)
- [ ] 等待队列按添加顺序清晰显示
- [ ] 可以单击移除队列中的任何发言者
- [ ] 切换到下一位发言者时, 计时自动重置并开始
- [ ] 队列为空时, 系统有明显提示

#### 故事2: 处理动议

**作为** Chair
**我想要** 系统帮我记录动议并引导我完成处理流程
**以便** 不遗漏动议且处理顺序正确

**场景描述**:
1. 会议正在进行 GSL, 某代表举手提出 Motion for Moderated Caucus
2. Chair 在系统中记录该动议: 类型=Moderated Caucus, 总时长=10分钟, 单次发言=1分钟, 提出者=Country A
3. 系统提示 Chair: "需要附议, 请询问是否有附议"
4. 有代表附议, Chair 点击"有附议"
5. 系统提示 Chair: "请对该动议进行表决"
6. Chair 进入表决流程, 记录票数
7. 表决通过, 系统自动将会议状态切换到 Moderated Caucus, 并启动 10 分钟倒计时

**验收标准**:
- [ ] 可以在 30 秒内记录一个完整的动议信息
- [ ] 系统明确提示当前需要做什么(询问附议/进入表决/执行动议)
- [ ] 表决通过后, 自动切换会议状态且启动对应计时
- [ ] 如果有多个动议, 系统按优先级排序并提示应先处理哪个
- [ ] 动议历史记录清晰可查

#### 故事3: 进行表决

**作为** Chair
**我想要** 快速准确地统计表决结果
**以便** 不出错且节省时间

**场景描述**:
1. Chair 需要对一个动议进行表决
2. Chair 宣布"请赞成的代表举手", 快速数数, 在系统中输入"15"
3. Chair 宣布"请反对的代表举手", 输入"8"
4. Chair 宣布"请弃权的代表举手", 输入"2"
5. 系统自动计算: 总票数=25, 赞成率=60%, 判定为"通过"(简单多数)
6. Chair 宣布结果, 系统记录该表决

**验收标准**:
- [ ] 可以在 10 秒内完成票数输入
- [ ] 系统自动计算总票数和赞成率
- [ ] 明确显示表决结果(通过/未通过)
- [ ] 可以修改已输入的票数(如数错了)
- [ ] 表决结果自动关联到对应的动议
- [ ] 支持不同表决规则(简单多数/绝对多数/全体一致)

#### 故事4: 管理 Moderated Caucus

**作为** Chair
**我想要** 在 Moderated Caucus 中同时管理总时长和单次发言时长
**以便** 不超时且发言分配合理

**场景描述**:
1. Moderated Caucus 通过: 总时长 10 分钟, 单次发言 1 分钟
2. 系统显示: 总倒计时 10:00, 当前发言倒计时 1:00
3. 有 3 个代表举手, Chair 添加到队列
4. 第一位代表开始发言, 单次计时开始: 0:59, 0:58...
5. 1 分钟到, 系统提示, Chair 让下一位发言
6. 单次计时重置为 1:00, 总计时继续: 9:05, 9:04...
7. 总时长到达 0:00, 系统明显提示"Moderated Caucus 结束", 自动切换回 GSL

**验收标准**:
- [ ] 同时显示总倒计时和单次发言倒计时, 且互不干扰
- [ ] 单次发言结束时, 单次计时重置, 总计时继续
- [ ] 总时长到达时, 自动提示并切换会议状态
- [ ] 可以提前结束 Moderated Caucus(如无人发言)
- [ ] 可以延长 Moderated Caucus 时间(如需要)

#### 故事5: 新手 Chair 第一次使用

**作为** 新手 Chair
**我想要** 不看文档就能快速上手使用系统
**以便** 在真实会议中能够信任并依赖它

**场景描述**:
1. Chair 第一次打开系统, 看到清晰的界面布局
2. 界面上有明确的区域: 当前状态, 发言队列, 动议列表, 计时器
3. 每个按钮都有清晰的文字说明(如"添加发言者", "记录动议", "开始表决")
4. Chair 点击"开始会议", 系统默认进入 GSL 状态
5. Chair 尝试添加发言者, 系统有即时反馈(发言者出现在队列中)
6. Chair 尝试记录动议, 系统有表单引导(填写动议类型, 时长等)
7. Chair 完成一次完整的动议处理流程, 感到清晰和可控

**验收标准**:
- [ ] 界面布局清晰, 一眼能看懂当前会议状态
- [ ] 所有操作按钮文字清晰, 无歧义
- [ ] 每个操作有即时反馈(不让用户猜测是否成功)
- [ ] 关键流程有引导提示(如"下一步需要表决")
- [ ] 错误操作有明确提示(如"当前状态不允许该操作")
- [ ] 新手 Chair 在 5 分钟内能完成基本操作

## 6. 功能详细说明

### 6.0 Roll Call (会前点名)

#### 核心作用

Roll Call 不仅是点名行为, 而是后续表决规则, 生效票数与会议合法性判断的基础. 该模块让 Chair 在会议开始前, 清晰, 准确, 可记录地完成这一关键步骤.

**核心价值**:
- 确认各代表团是否出席会议
- 明确 Present / Present and Voting 等状态
- 自动建立会议的有效代表列表
- 为后续简单多数, 三分之二多数等表决计算提供基数

#### 流程设计

**阶段1: 代表列表输入**

输入方式:
- **批量粘贴** (推荐):
  - 提供大文本框
  - 每行一个国家/代表团名称
  - 例如:
    ```
    United States of America
    China
    Russian Federation
    United Kingdom
    France
    ```
  - 系统自动解析为代表列表

- **单个添加**:
  - 输入框 + "Add" 按钮
  - 逐个添加代表名称
  - 适合小规模会议或临时添加

显示:
- 显示已输入的代表列表
- 按字母顺序或输入顺序排列
- 每个代表旁有删除按钮(如输入错误)

**阶段2: Roll Call 执行**

Chair 点击 "Start Roll Call", 进入点名流程:

显示模式:
- 列表显示所有代表
- 每个代表旁有三个状态按钮:
  - **Present** (P)
  - **Present and Voting** (PV)
  - **Absent** (A)
- 点击即可标记状态
- 已标记的代表有视觉反馈(如变色, 打勾)

快速操作:
- "Mark All as Present" 按钮(批量标记)
- "Mark All as Present and Voting" 按钮
- 然后逐个调整缺席的代表

**阶段3: 实时统计**

在 Roll Call 过程中, 实时显示统计数据:

```
Total Delegates: 15
Present: 12
Present and Voting: 10
Absent: 3
```

提示:
- 如果出席人数过少(如少于总数的 50%), 给出警告
- "Roll Call Complete?" 提示, 让 Chair 确认是否完成

**阶段4: 完成与锁定**

Chair 确认 Roll Call 完成, 点击 "Complete Roll Call":

- 系统锁定出席列表
- 显示确认信息:
  ```
  Roll Call Completed
  Total Present: 12
  Total Present and Voting: 10

  This will be used as the voting base for this session.
  ```
- 自动记录 Roll Call 结果和时间戳
- 进入会议准备状态(可开始 GSL)

会中修改:
- 如有代表迟到, Chair 可以在设置中"Update Roll Call"
- 需要明确操作(防止误操作)
- 修改后自动更新表决基数

#### 界面要求

- Roll Call 界面清晰, 适合投影显示
- 代表名称大字体, 状态按钮明显
- 统计数据实时更新, 显眼位置
- 操作简单: 单击即可标记状态
- 支持快速批量操作(Mark All)

#### 数据使用

Roll Call 完成后, 数据用于:

1. **发言者添加**:
   - 搜索栏自动建议时, 优先显示出席代表
   - 输入 2 个字母触发智能建议

2. **表决计算**:
   - 简单多数 = (赞成票 > 反对票) 且 (赞成票 > 有效票数 / 2)
   - 绝对多数 = 赞成票 > (Present + Present and Voting) * 2/3
   - 系统自动应用正确的计算规则

3. **会议记录**:
   - Roll Call 结果作为会议记录的第一部分
   - 可导出为文本或包含在会议报告中

#### Present vs Present and Voting

**Present**:
- 代表出席会议
- 在程序性动议(Procedural Motions)表决中可以选择弃权
- 例如: Motion for Moderated Caucus, Motion to Suspend Meeting

**Present and Voting**:
- 代表出席且承诺在所有表决中投票(不弃权)
- 在实质性表决(Substantive Voting)中必须投赞成或反对
- 通常用于正式决议草案表决

**系统处理**:
- MVP 阶段暂不强制区分程序性和实质性表决
- 允许所有代表在任何表决中弃权
- 但记录 Present 和 Present and Voting 状态, 用于后续迭代

### 6.1 会议状态管理

#### 支持的会议状态

| 状态 | 英文 | 说明 | 允许的操作 |
|------|------|------|-----------|
| 总发言名单 | General Speakers List (GSL) | 默认会议状态, 代表按顺序发言 | 添加发言者, 记录动议 |
| 有主持核心磋商 | Moderated Caucus | 主席主持的结构化讨论, 有总时长和单次发言时长限制 | 添加发言者, 管理发言队列, 计时 |
| 自由磋商 | Unmoderated Caucus | 代表自由讨论, 无发言顺序 | 仅计时, 无发言管理 |
| 表决程序 | Voting Procedure | 对决议草案或重要动议进行正式表决 | 记录表决, 计算结果 |
| 会议暂停 | Suspension of Meeting | 会议临时暂停(如休息, 午餐) | 仅显示状态, 无其他操作 |

#### 状态切换规则

```
GSL → Moderated Caucus (通过动议)
GSL → Unmoderated Caucus (通过动议)
GSL → Voting Procedure (提出表决)
GSL → Suspension (通过动议)

Moderated Caucus → GSL (时间到或提前结束)
Moderated Caucus → Unmoderated Caucus (通过动议, 较少见)

Unmoderated Caucus → GSL (时间到)
Unmoderated Caucus → Moderated Caucus (通过动议, 较少见)

Voting Procedure → GSL (表决完成)

Suspension → GSL (恢复会议)
```

#### 界面要求

- 当前状态需要在界面最显眼位置(如顶部, 大字体, 高亮)
- 状态切换时有动画或过渡效果, 让 Chair 明确感知到变化
- 每次状态切换自动记录时间戳, 用于会后分析

### 6.2 发言队列管理

#### 添加发言者

**智能搜索 + 自动建议** (核心交互方式):

交互流程:
1. Chair 点击 "Add Speaker" 区域或输入框
2. 输入框获得焦点, 等待输入
3. Chair 开始输入代表名称(如 "US")
4. **输入 2 个字母后**, 系统自动弹出建议列表:
   - 显示匹配的代表名称(前缀匹配或包含匹配)
   - **仅显示 Roll Call 中标记为 Present 或 Present and Voting 的代表**
   - 建议列表按匹配度排序(前缀匹配优先)
   - 例如输入 "US":
     ```
     - United States of America
     - Russian Federation (包含 "us")
     ```
5. Chair 可以:
   - 点击建议列表中的某项, 直接选择
   - 继续输入完整名称后按回车
   - 输入不在 Roll Call 列表中的名称(允许, 处理特殊情况)

建议列表设计:
- 下拉显示, 最多显示 5 条建议
- 每条建议显示完整代表名称
- 当前高亮项(键盘上下键可切换)
- 点击或回车确认选择

**允许手动输入**:
- 如果输入的名称不在 Roll Call 列表中, 仍允许添加
- 用于处理:
  - 迟到的代表(尚未更新 Roll Call)
  - 观察员或特殊发言者
  - 临时情况

**显示方式**:
- 当前发言者: 大卡片, 高亮显示, 包含代表名称和剩余时间
- 等待队列: 列表显示, 按顺序编号(1, 2, 3...)
- 队列长度提示: 如 "Queue: 5 speakers"

#### 发言计时

**计时显示**:
- 大字体显示剩余时间(如 "0:45")
- 颜色变化:
  - 正常: 白色/绿色
  - 剩余 10 秒: 红色 + 闪烁
  - 超时: 强烈视觉提示
- 进度条: 可选, 可视化时间流逝

**计时控制**:
- 自动开始: 点击"开始发言"后自动倒计时
- 暂停/继续: 单一按钮切换
- 重置: 重置为初始时长
- 延长: 快速添加时间(如 +30 秒)
- 结束: 手动结束当前发言, 自动切换到下一位

**10 秒警报提示** (重点):
- 触发时机: 倒计时剩余 10 秒时
- 视觉提示: 计时器颜色变红 + 闪烁效果
- 听觉提示: 警报提示音(可静音)
- 设计目的: 确保 Chair 无论在看代表还是看屏幕都能注意到
- 不强制停止: Chair 可以选择让代表继续, 或手动结束

#### 队列操作

- 移除: 点击发言者旁的删除按钮
- 调整顺序: 拖拽或上下移动按钮
- 清空队列: 一键清空所有等待者(需二次确认)

### 6.3 动议处理

#### 常见动议类型

| 动议类型 | 参数 | 是否需要附议 | 是否需要表决 |
|---------|------|------------|------------|
| Motion for Moderated Caucus | 总时长, 单次发言时长, 议题 | 是 | 是 |
| Motion for Unmoderated Caucus | 时长 | 是 | 是 |
| Motion to Close Debate | 无 | 是 | 是(需绝对多数) |
| Motion to Adjourn the Meeting | 无 | 是 | 是 |
| Motion to Suspend the Meeting | 时长 | 是 | 是 |
| Motion to Set Speaking Time | 新的发言时长 | 是 | 是 |

#### 动议记录流程

1. **记录动议信息**
   - Chair 点击"记录动议"
   - 选择动议类型(下拉菜单)
   - 填写必要参数(如时长)
   - 填写提出者(可选)
   - 点击"提交"

2. **系统引导流程**
   - 如果需要附议: 提示"请询问是否有附议", 提供"有附议"/"无附议"按钮
   - 如果无附议: 动议失败, 状态变为 Failed
   - 如果有附议: 提示"请对该动议进行表决", 引导进入表决流程

3. **表决后执行**
   - 如果表决通过: 自动执行动议(切换会议状态, 启动计时等)
   - 如果表决未通过: 动议失败, 保持当前状态

#### 动议显示

- 待处理动议: 列表显示, 包含类型, 提出者, 状态
- 动议优先级: 如果有多个动议, 按优先级排序, 高优先级高亮
- 动议历史: 已处理的动议可折叠显示, 包含结果和时间

### 6.4 表决管理

#### 表决流程

1. **发起表决**
   - 来源1: 处理动议时自动进入
   - 来源2: Chair 手动点击"开始表决"(用于其他需要表决的事项)

2. **记录票数**
   - 显示三个输入框/计数器:
     - 赞成 (For)
     - 反对 (Against)
     - 弃权 (Abstain)
   - 输入方式:
     - 直接输入数字
     - 点击 +/- 按钮
     - 键盘快捷键(如 F/A/X)

3. **计算结果** (基于 Roll Call 数据)
   - 自动计算:
     - 总票数 = 赞成 + 反对 + 弃权
     - 有效票数 = 赞成 + 反对 (弃权不计入)
     - 表决基数 = Roll Call 中 Present + Present and Voting 的总数
   - 判定规则(根据动议类型自动应用):
     - **简单多数** (Simple Majority):
       - 条件: 赞成 > 反对
       - 用于: 程序性动议(Moderated Caucus, Unmoderated Caucus, 等)
     - **绝对多数** (Absolute Majority):
       - 条件: 赞成 > 表决基数 * 50%
       - 用于: 某些特殊动议(Close Debate, 等)
     - **三分之二多数** (Two-thirds Majority):
       - 条件: 赞成 >= 表决基数 * 2/3
       - 用于: 实质性决议(Substantive Resolutions)
     - **全体一致** (Unanimity):
       - 条件: 反对 = 0
       - 用于: 特殊情况
   - 显示内容:
     - 票数明细: For: X, Against: Y, Abstain: Z
     - 表决基数: Voting Base: N (来自 Roll Call)
     - 通过阈值: Required: X votes (根据规则计算)
     - 结果: "Pass" (通过) 或 "Fail" (未通过)
   - 合法性检查:
     - 如果总票数 > 表决基数, 显示警告(可能有代表重复投票或统计错误)
     - 如果总票数明显少于表决基数, 提示可能有代表未投票

4. **确认并执行**
   - Chair 确认结果无误, 点击"确认"
   - 系统记录该表决, 并执行对应动作(如切换会议状态)

#### 表决历史

- 显示已完成的表决列表
- 包含信息:
  - 表决事项(如"Motion for Moderated Caucus")
  - 票数明细(赞成/反对/弃权)
  - 结果(通过/未通过)
  - 时间戳
- 可导出或打印

### 6.5 计时功能

#### 计时器类型

1. **发言计时器**
   - 用途: 单次发言时长控制
   - 默认时长: 根据会议状态确定(GSL 默认 1 分钟, Moderated 根据动议设定)
   - 显示位置: 当前发言者旁边或独立大显示

2. **会议阶段计时器**
   - 用途: Moderated Caucus 或 Unmoderated Caucus 总时长控制
   - 显示位置: 顶部或侧边, 与发言计时器并列
   - 时间到: 自动切换回 GSL

3. **会议总时长计时器(可选)**
   - 用途: 显示会议已进行时长
   - 显示位置: 不显眼位置, 仅供参考

#### 计时器控制

- 所有计时器支持: 开始, 暂停, 继续, 重置, 延长
- 暂停会议时, 所有计时器自动暂停
- 恢复会议时, 计时器自动继续

#### 时间提示

- 视觉提示:
  - 颜色变化(绿 → 黄 → 红)
  - 闪烁(最后 10 秒)
  - 进度条动画
- 听觉提示:
  - 时间到: 响铃或提示音
  - 剩余 30 秒: 可选预警音
  - 可全局静音或单独控制

## 7. 非功能性需求

### 7.1 性能要求

- **响应速度**: 所有操作(添加发言者, 记录动议, 表决等)响应时间 < 200ms
- **计时精度**: 倒计时误差 < 1 秒
- **并发处理**: 支持 Chair 快速连续操作(如连续添加多个发言者)
- **数据保存**: 每次操作自动保存, 刷新页面后状态不丢失

### 7.2 可靠性要求

- **容错性**: 如果 Chair 误操作(如错误添加发言者), 可以快速撤销
- **数据持久化**: 会议数据本地存储(LocalStorage), 防止浏览器崩溃导致数据丢失
- **离线支持**: 考虑离线模式(PWA), 网络断开时仍可使用基本功能
- **崩溃恢复**: 页面刷新或意外关闭后, 可自动恢复到最后状态

### 7.3 易用性要求

- **学习成本**: 新手 Chair 在 5 分钟内能完成基本操作(添加发言者, 记录动议, 表决)
- **操作效率**: 高频操作(如添加发言者)可在 5 秒内完成
- **视觉清晰度**: 界面布局清晰, 关键信息(当前状态, 当前发言者, 剩余时间)一眼可见
- **错误预防**: 关键操作(如清空队列, 结束会议)需要二次确认
- **即时反馈**: 每个操作有明确的视觉或听觉反馈

### 7.4 兼容性要求

- **浏览器**: 支持 Chrome, Edge, Safari, Firefox 最新版本
- **设备**: 优化笔记本电脑显示(13-15 寸屏幕), 横屏为主
- **分辨率**: 支持 1366x768 及以上分辨率
- **投屏**: 界面在投影仪上清晰可读(如果需要投屏给代表看)

### 7.5 扩展性要求

- **模块化设计**: 各功能模块(发言管理, 动议管理, 表决管理)相对独立, 便于后续扩展
- **规则配置**: 表决规则, 动议类型等可通过配置调整, 不需要修改代码
- **数据导出**: 会议数据结构清晰, 便于后续导出和分析

### 7.6 界面与交互要求(MVP 关键决策)

#### 投屏方式
- **决策**: 投屏是 Chair 操作界面的镜像
- **实现**: 只需要设计一套界面, 投屏直接镜像显示
- **要求**: 界面必须适合投影显示(大字体, 高对比度, 清晰布局)

#### 交互方式
- **决策**: MVP 只支持鼠标点击, 不支持快捷键
- **原因**: 降低学习成本, 保持简单直观
- **要求**: 高频操作不超过 2 次点击

#### 代表名单
- **决策**: MVP 不支持预设代表名单
- **实现**: 手动输入代表名称(如 "USA", "China", "Country A")
- **原因**: 简化会前设置, 保持 MVP 轻量

#### 语言
- **决策**: 全界面使用英文
- **原因**: 符合模拟联合国会议的真实使用场景
- **要求**: 所有按钮, 提示, 标签使用英文

#### 视觉主题
- **决策**: 使用浅色主题
- **原因**: 投影显示效果更好, 会场光线下更清晰
- **配色要求**:
  - 背景: 白色或浅灰色(#FFFFFF, #F5F5F5)
  - 文字: 深色(#333333, #000000)
  - 高对比度, 确保投影清晰可读
  - 状态提示使用明显颜色(绿色=正常, 红色=警告)

## 9. 成功与失败标准

### 9.1 MVP 成功标准

#### 定量指标

1. **使用意愿**:
   - 目标: 至少 5 位不同经验水平的 Chair 在真实会议中使用
   - 底线: 至少 3 位

2. **错误率**:
   - 目标: 流程性错误(漏掉发言者, 动议处理错误, 表决计算错误)比使用文档时下降 50% 以上
   - 底线: 下降 30%

3. **操作效率**:
   - 目标: 常用操作(添加发言者, 记录动议, 表决)比文档方式快 50%
   - 底线: 快 30%

4. **重复使用率**:
   - 目标: Chair 在第一次使用后, 第二次会议仍愿意使用
   - 底线: 50% 的 Chair 愿意再次使用

#### 定性指标

1. **信任度**:
   - Chair 感到"系统可靠, 我可以依赖它"
   - Chair 不需要同时维护纸质/文档备份

2. **认知负担**:
   - Chair 感到"我不需要记住太多信息, 系统帮我管理了"
   - Chair 可以更专注于判断和决策

3. **专业性感知**:
   - 代表或观察者感到"这个 Chair 很专业, 流程很清晰"
   - 组织方感到"会议质量更可控"

4. **学习成本**:
   - 新手 Chair 在 5 分钟内能基本掌握
   - 不需要复杂的培训或文档

### 9.2 失败标准(立即停止信号)

#### 致命问题

1. **增加负担**:
   - Chair 反馈"用系统比用文档更麻烦"
   - Chair 需要同时操作系统和维护备份, 工作量翻倍

2. **不可靠**:
   - 系统在会议中崩溃或卡顿
   - 数据丢失或计算错误
   - Chair 不敢在真实会议中使用

3. **学习成本过高**:
   - 新手 Chair 花了 30 分钟仍不会用
   - 需要复杂的培训才能上手
   - Chair 试用后放弃

4. **无价值**:
   - Chair 反馈"这个系统没有解决我的痛点"
   - 错误率没有明显下降
   - 效率没有明显提升

#### 警告信号

1. **使用率低**:
   - Chair 打开系统后很快关闭, 继续用文档
   - Chair 只在练习时用, 真实会议不敢用

2. **功能不匹配**:
   - Chair 频繁需要跳出系统手动处理
   - 系统流程与真实议事规则不符

3. **体验差**:
   - 界面混乱, Chair 找不到需要的功能
   - 操作不流畅, 频繁误操作
   - 提示不清晰, Chair 不知道下一步该做什么

### 9.3 验证方案

#### 验证方式

1. **真实会议测试**:
   - 找 5 位不同经验水平的 Chair(2 位新手, 3 位有经验)
   - 在真实或模拟会议中使用系统
   - 全程录像并记录错误和卡顿

2. **数据收集**:
   - 系统自动记录操作日志(点击次数, 操作时长, 错误次数)
   - Chair 会后填写问卷(信任度, 认知负担, 使用意愿)
   - 观察者评分(会议流畅度, 专业性感知)

3. **对比实验**:
   - 同一 Chair 先用文档, 再用系统, 对比数据
   - 或不同 Chair 分别用文档和系统, 对比结果

#### 时间表

- **Week 1-2**: 开发 MVP
- **Week 3**: 内部测试, 修复明显 Bug
- **Week 4**: 找 2-3 位 Chair 进行小规模测试
- **Week 5**: 收集反馈, 快速迭代
- **Week 6**: 扩大到 5 位 Chair, 真实会议验证
- **Week 7**: 数据分析, 决策是否继续

## 11. 技术建议(供开发参考)

### 11.1 技术栈建议

- **前端框架**: React / Vue 3 (组件化, 状态管理清晰)
- **状态管理**: Zustand / Pinia (轻量, 易用)
- **UI 组件库**: Ant Design / Element Plus (开箱即用, 专业)
- **数据持久化**: LocalStorage (简单可靠) + IndexedDB (大数据量)
- **PWA**: Workbox (支持离线和安装)
- **音频提示**: Web Audio API / Howler.js

### 11.2 数据结构建议

```typescript
// 代表(用于 Roll Call)
interface Delegate {
  id: string
  name: string // 如 "United States of America"
  attendance: 'present' | 'present_and_voting' | 'absent'
  timestamp: Date // 标记时间
}

// Roll Call 结果
interface RollCallResult {
  delegates: Delegate[]
  totalDelegates: number
  presentCount: number
  presentAndVotingCount: number
  absentCount: number
  completed: boolean
  completedAt?: Date
}

// 会议状态
interface MeetingState {
  id: string
  name: string
  status: 'setup' | 'roll_call' | 'GSL' | 'Moderated' | 'Unmoderated' | 'Voting' | 'Suspension'
  startTime: Date
  currentTime: Date
  rollCall: RollCallResult
}

// 发言者
interface Speaker {
  id: string
  name: string
  status: 'speaking' | 'waiting'
  speakingTime: number // 秒
  remainingTime: number // 秒
}

// 动议
interface Motion {
  id: string
  type: string
  proposer: string
  parameters: Record<string, any> // 如 { totalTime: 600, speakingTime: 60 }
  status: 'pending' | 'in_voting' | 'passed' | 'failed'
  voteResult?: VoteResult
  timestamp: Date
}

// 表决结果
interface VoteResult {
  for: number
  against: number
  abstain: number
  result: 'pass' | 'fail'
  timestamp: Date
}
```

### 11.3 关键技术点

1. **精确计时**: 使用 `requestAnimationFrame` 或 `setInterval(100ms)` 更新倒计时, 避免误差累积
2. **数据持久化**: 每次状态变化自动保存到 LocalStorage, 页面加载时恢复
3. **音频提示**: 预加载音频文件, 确保提示音即时响应
4. **响应式设计**: 适配不同屏幕尺寸, 特别是投影仪显示
5. **性能优化**: 虚拟列表(如果发言队列很长), 防抖节流(频繁操作)

| 特权动议 | Privileged Motion | 优先级最高的动议(如中止会议) |
| 程序动议 | Procedural Motion | 关于会议流程的动议(如有主持核心磋商) |
